epub-audiobook

Living specification for a CLI that converts an EPUB into a single audiobook MP3 with navigable chapter markers, using Google Cloud Text-to-Speech Neural2 voices.

Status: Working tool Version: 1.0Updated: 2026-06-11 Owner: @shubhambakreType: Python CLI

01 Executive summary

A single-purpose Python CLI that turns an EPUB into one MP3 audiobook with ID3 chapter markers, synthesized with Google Cloud TTS Neural2 voices. It is engineered for speed and cost on long books: SSML is packed into ~4.8 KB sentence-aligned chunks, synthesized in parallel under a rate limiter, and concatenated with ffmpeg -c copy (no re-encode). Runs are resumable via a per-chunk cache.

◆ Decision
Emit ID3 CHAP/CTOC frames in MP3 rather than producing M4B — chapter nav works in Apple Books/VLC without the heavier container.

02 Problem statement

Common EPUB→MP3 scripts re-encode every chunk through pydub, run sequentially, and emit tiny SSML payloads — slow and expensive on book-length text, often with no chapter navigation.

▲ Pain addressed
Hours of needless re-encoding and serial API calls, plus audiobooks with no way to jump between chapters.

03 Goals

● G1 — Fast
Parallel synthesis + -c copy concat; no re-encode of audio.
● G2 — Navigable
Real chapter markers via ID3 CHAP/CTOC.
● G3 — Cheap & predictable
--dry-run reports exact character counts before spending TTS quota.
● G4 — Resumable
Per-chunk cache keyed by chapter+index survives interruptions.

04 Non-goals

  • Not a GUI or a hosted service — it is a local CLI.
  • Not a multi-provider TTS abstraction (Google Neural2 only).
  • Not a DRM remover; input must be a readable EPUB.
  • Does not produce M4B/M4A by design.

09 Architecture overview

A single module (audiobook.py) orchestrating a linear pipeline with a parallel synthesis stage.

EPUB→parse chapters→SSML chunker→parallel TTS (rate-limited)→ffmpeg concat→ID3 chapters→out.mp3

10 Data model & interfaces

CLI interface
python audiobook.py INPUT.epub OUT.mp3 [--voice ID] [--speaking-rate R]
                    [--dry-run] [--workdir DIR] [--keep-workdir]
External interface — Google Cloud TTS

Auth: GOOGLE_APPLICATION_CREDENTIALS (service account, recommended) or GOOGLE_TTS_API_KEY. Output: per-chunk MP3 bytes, cached on disk keyed by chapter+index.

Document the exact cache filename scheme and the SSML emphasis rules.

11 Main workflows

W1 — Estimate then synthesize

--dry-run to see char count/cost → real run → resumes from cache if interrupted.

W2 — Tune voice/rate

Re-run with --voice / --speaking-rate; cached chunks for unchanged settings are reused.

13 Testing & validation

Primary validation is --dry-run on representative EPUBs (counts + chunk boundaries), plus spot-checking chapter navigation in a player.

? Gap
No automated test suite yet.

Add unit tests for the SSML chunker (boundary/size) and a golden-file test for ID3 chapter frames.

14 Monitoring & operations

  • Progress via tqdm bars during synthesis.
  • Cost visibility via --dry-run char counts.
  • Failure recovery via the resumable cache.

Define behavior on partial API failures (retry/backoff policy) and a max-cost guardrail.

15 Risks & assumptions

▲ R1 — Cost overrun
Large books exceed the free tier. Mitigation: mandatory --dry-run habit.
▲ R2 — Credential leakage
API keys in URLs/commits. Mitigation: prefer service accounts; gitignore secrets.
▲ R3 — Player compatibility
Some players ignore ID3 chapters. Mitigation: document tested players (Apple Books, VLC).

Assumptions: input is a non-DRM EPUB; ffmpeg on PATH; English/European Neural2 voice.
